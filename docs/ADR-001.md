# ADR-001: AttractorFlow System Architecture

**Status:** Proposed
**Date:** 2026-03-26
**Deciders:** Attractor Project core team

---

## Context

Agentic AI orchestration systems (Claude Code, AutoGPT, LangGraph, etc.) exhibit emergent dynamical behaviors — repetition loops, sycophantic drift, context divergence, mode collapse — that current tooling treats as isolated bugs. Recent peer-reviewed research (2024–2025) establishes these behaviors are classifiable attractors in a formal dynamical systems sense. Five independent papers (Tacheny et al., Wang et al., Ramsauer et al., Bhargava et al., DMET framework) have demonstrated this analytically. No system yet uses this knowledge *prescriptively* to design or monitor agent orchestration.

We must decide:
1. **How to represent agent state** for dynamical analysis (embedding strategy)
2. **How to compute stability metrics** in real time without model retraining
3. **Where to run the monitor** relative to the agent loop (in-process vs. sidecar vs. MCP)
4. **What design patterns** govern the orchestration layer
5. **What language/framework** to implement the MCP server in

These decisions interact: the embedding strategy determines computational cost, which influences whether we run in-process or as a sidecar, which constrains the protocol.

**Forces at play:**
- Must add zero model training cost (pure harness innovation)
- Must integrate with Claude Code's existing MCP + subagent architecture
- Lyapunov computation must be fast enough not to break agent flow (< 200ms per step)
- System must be interpretable: engineers must understand what the metrics mean
- Must run locally on developer laptops (no GPU assumed)

---

## Decision

We implement AttractorFlow as a **local Python MCP server (stdio transport) using FastMCP**, with **sentence-transformers (all-MiniLM-L6-v2)** for embedding, **numpy/scipy** for trajectory statistics, and a **layered design pattern** (Monitor → Estimator → Classifier → Orchestrator) that mirrors the separation of concerns in classical control theory.

The orchestration layer uses **Claude Code's native subagent system** (markdown-defined agents) rather than a bespoke workflow engine, delegating regime-specific behavior to specialist agents (explorer, convergence, meta-orchestrator).

## Options Considered

### Option A: In-Process Python Library (Chosen Architecture Variant)

Embed the phase space monitor as a library imported directly into the agent's Python process.

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low |
| Latency | Very low (< 5ms) |
| Isolation | None — crashes affect agent |
| Claude Code integration | Poor (Claude Code doesn't expose Python runtime) |
| Multi-agent support | None |

**Pros:** Simplest, lowest latency
**Cons:** Not compatible with Claude Code's sandboxed execution model. Claude agents run as separate processes; there is no shared Python runtime to import into.

**Verdict: Eliminated.** Claude Code's architecture makes this infeasible.

---

### Option B: Sidecar HTTP Service (REST/gRPC)

Run AttractorFlow as a local HTTP server. Agent makes HTTP calls at each step.

| Dimension | Assessment |
|-----------|------------|
| Complexity | Medium |
| Latency | Low (< 30ms for local HTTP) |
| Isolation | Good — independent process |
| Claude Code integration | Manual — requires agent to know the URL |
| Multi-agent support | Natural — multiple agents call same endpoint |
| Authentication | Not needed for local, but adds complexity for remote |

**Pros:** Language-agnostic, multi-agent support, natural REST semantics
**Cons:** Requires agents to discover service URL; not composable with Claude Code's tool-use system; developers must manually wire the HTTP calls; no native tooling support.

**Verdict: Viable but suboptimal.** The MCP protocol is strictly better for Claude Code integration.

---

### Option C: Claude Code MCP Server — stdio transport ✅ CHOSEN

Expose AttractorFlow as an MCP server over stdio. Claude Code's `.mcp.json` config registers it; all agents in the project get the tools automatically.

| Dimension | Assessment |
|-----------|------------|
| Complexity | Medium |
| Latency | Low (< 100ms including embedding) |
| Isolation | Good — subprocess |
| Claude Code integration | Native — tools appear in Claude's tool list automatically |
| Multi-agent support | Via shared MCP server state (trajectory buffer) |
| Authentication | Not needed for local stdio |
| Discoverability | Automatic via Claude Code tool listing |

**Pros:**
- Tools appear natively in Claude's tool-use paradigm — no custom wiring needed
- `.mcp.json` in project root enables zero-config adoption
- Subagents all share one MCP server process → shared trajectory state
- MCP is Anthropic's officially supported extension mechanism
- stdio transport eliminates network surface area and auth complexity
- FastMCP reduces boilerplate to decorators and Pydantic models

**Cons:**
- MCP protocol overhead (JSON-RPC) vs. direct function calls
- Server must maintain trajectory state across tool calls (requires in-memory buffer)
- One server process per project (not per agent) — needs careful state partitioning for multi-agent

**Verdict: Adopted.**

---

### Option D: TypeScript MCP Server

Same as Option C but using the TypeScript MCP SDK.

| Dimension | Assessment |
|-----------|------------|
| ML library access | Poor — no sentence-transformers, no numpy |
| Embedding options | External API only (adds latency + cost) |
| Signal processing | Limited vs. Python's scipy |
| Team familiarity | Lower for ML-heavy work |
| Runtime startup | Faster (Node.js) |

**Pros:** Stronger TypeScript typing; faster startup; broader MCP ecosystem compatibility
**Cons:** The core functionality (embedding, Lyapunov computation, PCA, autocorrelation) is fundamentally a scientific Python workload. Embedding via external API would add 100-300ms per call and introduce a remote dependency — exactly what we're trying to avoid.

**Verdict: Eliminated.** Python is the right language for this computation profile.

## Design Patterns

### Actor-Critic Architecture (for Orchestration Layer)

The attractor-orchestrator implements an **Actor-Critic** loop drawn from reinforcement learning but adapted to the agentic context:

```
Actor (Explorer / Convergence subagent)
    │ generates states (code, plans, outputs)
    ▼
Critic (Phase Space Monitor + Classifier)
    │ evaluates trajectory quality (λ, regime)
    ▼
Orchestrator (attractor-orchestrator agent)
    │ policy: regime → action (continue / perturb / decompose / halt)
    ▼
Actor receives modified context + tool selection
```

The critic does not evaluate *output quality* (which would require domain knowledge) but *trajectory quality* — is the system converging, cycling healthily, exploring usefully, or drifting? This separation of concerns is the key innovation: domain-agnostic stability metrics enable a domain-agnostic orchestrator.

### Observer Pattern (for Phase Space Monitor)

The PhaseSpaceMonitor acts as an **Observable** that agents subscribe to indirectly via MCP calls. The trajectory buffer is an append-only event log; the classifier and Lyapunov estimator are observers that compute views over it on demand. No push notifications — pull-based by design.

### Strategy Pattern (for Regime-Specific Interventions)

Each regime maps to a Strategy object:

```python
class RegimeStrategy(Protocol):
    def recommend_action(self, trajectory: Trajectory) -> OrchestratorAction: ...
    def generate_perturbation(self, trajectory: Trajectory) -> str: ...
```

Strategies: `ConvergingStrategy`, `CyclingStrategy`, `ExploringStrategy`, `DivergingStrategy`, `StuckStrategy`, `OscillatingStrategy`. New regimes can be added without touching the classifier.

### Command Pattern (for Slash Commands)

`/attractor-status` and `/phase-portrait` are Claude Code slash commands that invoke AttractorFlow MCP tools and format the output for human consumption. They are composable, discoverable, and version-controlled alongside the agent definitions.

### Layered Architecture

```
┌─────────────────────────────────────────────┐
│  Layer 4: Orchestration                     │
│  attractor-orchestrator agent               │
│  explorer-agent, convergence-agent          │
├─────────────────────────────────────────────┤
│  Layer 3: Diagnosis                         │
│  AttractorClassifier, BifurcationDetector   │
├─────────────────────────────────────────────┤
│  Layer 2: Estimation                        │
│  LyapunovEstimator, BasinDepthEstimator      │
├─────────────────────────────────────────────┤
│  Layer 1: Measurement                       │
│  PhaseSpaceMonitor, SentenceEmbedder        │
├─────────────────────────────────────────────┤
│  Layer 0: Transport                         │
│  FastMCP, stdio, JSON-RPC                   │
└─────────────────────────────────────────────┘
```

Each layer depends only on the layer below it. The orchestration layer is replaceable (Claude Code today, another framework tomorrow) without touching layers 0–3.

## Trade-off Analysis

**Embedding model choice (MiniLM vs. larger model):**
MiniLM-L6-v2 (384 dims, 22MB) embeds in ~30ms on CPU vs. ~200ms for a larger model (768 dims). For a 50-step agent session, this saves 8.5 seconds. The dynamical analysis doesn't require semantic richness — it requires *trajectory geometry*, which is well-captured at 384 dims. The tradeoff favors speed.

**FTLE window size (W=8 default):**
Smaller W (4) → responsive but noisy; larger W (16) → stable but lags regime changes by ~8 steps. W=8 is a Goldilocks value confirmed by the Tacheny et al. framework: contractive vs. oscillatory regimes are distinguishable in as few as 5 steps.

**Shared state vs. per-agent buffers:**
The single MCP server process maintains one trajectory buffer shared across all agents in a session. This enables cross-agent bifurcation detection (seeing that two subagents are on divergent trajectories). The alternative — per-agent buffers — would lose this signal. The risk (state pollution between unrelated tasks) is mitigated by the `attractorflow_checkpoint()` tool which marks session boundaries.

**Stdio vs. Streamable HTTP transport:**
Stdio eliminates auth, network config, and port conflicts. For multi-user or remote scenarios, streamable HTTP is the upgrade path. The FastMCP transport API makes switching a one-line change.

## Consequences

**Becomes easier:**
- Adding new regime types — implement a new Strategy, register with classifier
- Swapping embedding models — PhaseSpaceMonitor wraps the embedder behind a protocol
- Multi-agent trajectory comparison — all agents write to the same buffer
- Debugging stuck agents — `/attractor-status` gives immediate diagnosis without log diving

**Becomes harder:**
- Stateless operation — the MCP server is stateful; Claude Code sessions that restart must re-seed the trajectory buffer
- Testing the classifier — requires realistic agent trajectories as fixtures, not just unit inputs
- Scaling to very long sessions — trajectory buffer grows linearly; must implement FIFO eviction

**Will need to revisit:**
- Embedding model selection once code-specific embeddings mature
- Per-agent vs. shared trajectory buffers as multi-agent use cases become clearer
- Whether BifurcationDetector needs a feedback loop to the embedding layer for sensitivity testing
